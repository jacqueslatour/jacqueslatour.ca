How Verification Works

From signing a document to verifying it in your browser — here is the complete trust chain.

Four Steps to Trust

1

Author Creates Content

A document, web page, certificate, or any file is created by its author.

2

Content Is Signed

TrustDID computes a SHA-256 hash and signs it with the author's Ed25519 + ML-DSA-44 keys, creating a .vrfy manifest.

3

Published Together

The file and its .vrfy manifest are published side by side. Anyone can access both.

4

Browser Verifies

The browser extension fetches the manifest, re-hashes the content, and verifies the cryptographic signatures against the signer's DID.

What the Browser Extension Shows

The TrustDID extension uses progressive trust — five color states tell you exactly how much you can trust a page.

Orange — Unverified

No .vrfy manifest found for this page. The page is not signed.

Yellow — Hash Verified, Different Domain

Content matches its manifest, but the signer's DID is on a different domain than the page. Informational — the document may be legitimately hosted elsewhere.

Blue — Hash Verified, Same Domain

Content matches its manifest and the signer's domain matches the page domain. Click "Advanced Verify" to fully verify the signature chain.

Green — Fully Verified

All checks passed: hash integrity, signature validity, DID resolution, DNSSEC, and Trust Registry. This is what you should see on every page of this site.

Red — Tampered or Invalid

The content does not match its signed manifest, or the signature verification failed. Do not trust this document.

Three Assurance Levels

Not all verification is equal. TrustDID provides layered assurance depending on the signer's infrastructure.

Standard

DID Document Only

  • Document hash integrity
  • Ed25519 signature valid
  • DID document resolved via HTTPS
  • Signer identity established
High

DID + DNSSEC

  • Everything in Standard
  • TLSA record verified
  • DNS integrity via DNSSEC anchor
  • Domain ownership proven cryptographically
Maximum

DID + DNSSEC + Trust Registry

  • Everything in High
  • Trust Registry authorization checked
  • Signer recognized by trust ecosystem
  • Highest confidence in signer identity

Inside a .vrfy Manifest

Every signed file has a companion manifest containing the hash, signer identity, and cryptographic proofs.

{
  "version": "3.0",
  "type": "VerifiableDocumentSignature",
  "did": "did:web:jacqueslatour.ca#active",
  "payload": {
    "documentHash": "a1b2c3d4e5f6...",         // SHA-256 of file
    "hashAlgorithm": "SHA-256",
    "fileName": "certificate.html",
    "timestamp": "2026-03-08T14:30:00Z"
  },
  "proof": {
    "type": "Ed25519Signature2020",       // Classical signature
    "proofValue": "base64..."
  },
  "manifestProof": {
    "type": "Ed25519Signature2020",       // Envelope integrity
    "proofValue": "base64..."
  },
  "pqManifestProof": {
    "type": "MLDSASignature2024",         // Post-quantum proof
    "proofValue": "base64..."
  }
}

What Gets Checked

✓ Document Integrity

SHA-256 hash of file matches manifest

✓ Manifest Integrity

Envelope signature prevents tampering

✓ DID Resolution

Signer's DID document fetched and parsed

✓ Signature Valid

Ed25519 signature verified against public key

✓ PQ Manifest Proof

ML-DSA-44 post-quantum signature verified

✓ DNSSEC Anchor

TLSA record matches DID public key

✓ Trust Registry

Signer authorized in trust ecosystem

✓ Not Expired

Signature within validity period
